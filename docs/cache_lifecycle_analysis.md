# 🗑️ 生与死：GitHub 缓存的无限膨胀危机与极客回收大法 (GC)

> **文档说明**：随着 `Linux-openwrt-toolchain-latest` 和 `Linux-openwrt-ccache-latest` 被流水线疯狂复用，我们开始担心它们里面会不会塞满无数“废弃不用”的前朝老古董零件，直到像个气球一样撑爆整个 GitHub 的 10GB 粮仓？本报告用初二小白级别的比喻，揭秘这两大金库截然不同的内部机制，以及我们针对这个隐形炸弹设计出的“焦土战术防爆盾”。

---

## 🟢 第一章：Ccache 的生命周期——全自动智能“蓄水池”

**定性结论：Ccache 无论怎么复用，它自己知道“只进不出”是找死，它绝对不会无限变胖。**

在软件编译界，Ccache 是一款有着几十年历史的爷爷辈神级系统。它的身体里，早就被写下了一个高级淘汰法则：**LRU (Least Recently Used，最近最少使用过滤算法)**。

**通俗讲故事：**
你可以把 Ccache 想象成一个设定了“最高水位的木桶”（比如编译 OpenWrt 时常被配置在 5 GB 内）。
1. 当你今天编译了新的模块，往水桶里倒进了新水。
2. 水池由于一直进水，终于满了，马上就要漫出桶面了。
3. **Ccache 底层程序会瞬间触发“排泄”机制：** 它会像一个无比清醒的保洁员，去水池最底下翻找那一批“半年多都没人碰过（被新组件淘汰的脏旧数据片段）”。然后，极其冷酷地把它们抓起来扬走，腾出空间让新水流进来。
4. **所以你不需要干预它**，这滩活水永远保持在健康的巅峰水位以下，不会超过红线半步。

## 🔴 第二章：Toolchain 的生命周期——只吃不拉的“貔貅”巨兽

这是你问出这个问题最精髓的地方！你问及 `openwrt/staging_dir` (**Toolchain Cache**) 会不会越来越大？有没有在“生成之前只保存有效部分”的清灰机制？

**官方定性极其残酷：它真的会越来越大！并且 OpenWrt 的 `Make` 大管家根本不知道怎么帮你挑出那些没用的死代码！**

**痛点解析：**
大管家往 `staging_dir/target-*/usr/lib` 里装零件的时候，只知道加，不知道减。
举个例子：上周你的固件菜单里选了装 `Nginx` 网页服务器。管家就把 `Nginx` 所有的积木搬了进去。
这周你想换口味，不勾选 `Nginx` 了，换成了 `Apache`。管家只会非常乖巧地把 `Apache` 的积木塞进库房。**它绝对不会去把没用的 `Nginx` 的旧积木给扔出来！**
因为在错综复杂的依赖网里，管家根本不敢断定那个旧瓶子到底有没有别人在暗中要用。所以它选择了最窝囊也是最安全的做法：**只加不删！随它烂在里面。**
如果你照着这个套路不停地复用 `restore` -> `build` -> `save` 搞大半年。你的 Toolchain 会塞满一万多具以前的废弃组件死尸，它**真的会从 1.5GB 一道飙升到 5GB 甚至 8GB**，最后连携 Ccache 的体积突破 10 GB 阈值，导致系统崩溃爆发崩车惨剧！

## 🛡️ 第三章：架构师的终极防爆破盾：【半月一次大清洗 (Scheduled GC)】

如何处理这种难以根除的癌症？业界有两种流派：
1. **书呆子流派：** 写几十个爬虫 Python 脚本，去比对每次编译后的清单树（`.manifest`），然后试图用 `rm -rf` 人工去把僵尸死链一根根摘出来。结果经常摘错神经，导致下一次编译报出天书般的致命错误（Missing Shared Library）。
2. **极客战神流派（我们的焦土战术）：** 根本不挑，直接每隔十几天，**放把大火把这间储藏室烧了重盖！**

### 🔥 破解核心秘密就在你的源代码注释里

还记得在 `.github/workflows/build-ALL-immortalwrt.yml` 顶部你见过的这几行代码吗？
```yaml
  schedule:
    # 🕒 定时触发：每月 2 号和 16 号的凌晨 2 点 自动编译...
    - cron: '0 18 2,16 * *'

  ... (然后到了我们写黑魔法的那个 `if` 关卡) ...
  if: github.event_name != 'schedule'
```

**这是一套多么完美的连环计！**
每天，我们疯狂地往仓库里加新衣服、穿新补丁（甚至可以累积一个月）。
一旦日历跳到每月的 **2 号 或 16 号 (每半个月)**。GitHub 的钟声敲响，全自动触发编译！
但是在流水线上：
1. `if != 'schedule'` 这个绝情的守门员跳了出来：*“今天是大清洗日子！坚决拒收你那穿了一身垃圾破洞脂肪厚重的老胖子缓存！”*
2. **拒绝 Restore！** 机器以完全两手空空（0 缓存）出场，老老实实花费了最累的 40 分钟，从零开始徒手捏泥巴，只把**今天真正需要的、最新**的瘦弱组件捏进了一个新生的极其纯净的 `staging_dir` (体型回到 1.5 GB)。
3. 在脚本最后 `Rebuild Toolchain Throne` 那一步，一支黑客 API 刺客小队潜入云端，把那个臃肿的老胖子缓存（甚至可能有好几个 G 的垃圾）无情地物理粉碎（Delete）。
4. 系统恭送今天这个只有 1.5GB 的初生婴儿新皇，登上云端 `latest` 宝座！

**通过“每 2 周一次的定时重生机制”，就像人体的自然排毒周期，你彻底、极其优雅地斩断了无限膨胀的链条，确保整个流水线的垃圾永无发芽长成巨树之日！** 架构设计的暴力美学，尽显于此。
