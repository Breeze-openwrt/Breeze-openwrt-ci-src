# 🧰 OpenWrt 深层原理：为什么 `tools` 不能/不敢缓存？

> **文档说明**：在实装了极限的 `Toolchain (staging_dir)` 缓存魔法后，系统依然会在每次构建初期，花费几分钟时间重跑 `make -C tools/... compile`。本文将深入 OpenWrt 的构建机制核心，通俗拆分这几分钟“无效劳动”背后的无奈与工程权衡。

---

## 🕵️‍♂️ 第一章：案发现场，为何明明有缓存却要重做？
你在盯着云端的暗黑控制台时，绝对发现了这样一个让人非常不爽的画面：
```bash
make[3] -C tools/mtd-utils compile
make[3] -C tools/mpc compile
...
```
我们在上一轮的“时空黑魔法”里，不仅装载了 `openwrt/staging_dir`，里面的 `staging_dir/host/` 其实就是这些 tools 编译成品存放大军的驻地！而且我们还加了 `sudo touch` 的时间戳欺骗大法！那**为什么监工还是要把它们重造一遍拿去覆盖现货？**

### 💡 核心原因：大管家不看仓库，只看“工地”（`build_dir`）
OpenWrt 的 `make` 系统判定一个工程干没干完，有一套非常诡异（甚至死板）的潜规则。

当它去生产一个像 `quilt`（打补丁工具）时：
1. 它会在 **临时工地区**（`openwrt/build_dir/host/quilt-xxx/`） 搭一个台子。
2. 在台子上叮叮当当敲出几千个中间的废木块（`.o` 编译产物文件）。
3. 把敲好的最终铁锤放入 **成品仓库**（`openwrt/staging_dir/host/bin`）。
4. **最致命的动作：** 它会在“临时工地里”放一个打卡记录签批单（通常是一个叫 `.built` 或 `.installed` 的 0 字节隐藏空文件）。

📌 **我们上一把黑魔法漏掉了什么？**
答案是：**我们故意漏掉了“临时工地 (`build_dir`)”！**
为了节约 GitHub 极度宝贵的 10GB 空间，我们只拿走了“成品仓库(`staging_dir`)”，把充满了废旧边角料的几十 GB 的“临时工地”给扔了。

于是，大管家一查账：*“成品仓库里的工具是不错，看着也挺新，但在隔壁临时工地里，我居然找不着之前开工的打卡签批单！没记录就是没干过，全部给我重干！”*
这就导致了这几分钟的重造。

---

## ⚖️ 第二章：要不要连它一起吞噬？(屠龙少年的抉择)

如果你有极其严重的 OCD（强迫症），连这 3~5 分钟都受不了，想彻底跳过 `tools` 的流水线，我们在代码层面的修改简直是一秒钟的事，只需要在 `build-ALL-immortalwrt.yml` 的 `Cache/Save` 步长长的一串后加上：
```yaml
      with:
        path: |
          openwrt/staging_dir
          openwrt/build_dir/host
          openwrt/build_dir/toolchain-*
        key: ...
```

但身为一名合格的架构师，这是一种**饮鸩止渴的作死行为**。

### 🚨 绝对不可跨越的死亡红线（风险预警）：
1. **体重爆炸 (Cache Bloating)**
   `build_dir/` 里面可是成千上万没被压缩的生肉废木材！一旦加上它，我们原本只有 **1 GB 多一点** 精瘦健康的 Toolchain 缓存，会像气球一样瞬间狂吞 **3 GB 到 5 GB** 以上的巨无霸空间！
2. **下载反而变慢 (I/O Bottleneck)**
   解压 1 GB 和解压 5 GB 的时长是完全不对等的，你很可能为了强行省下那 3 分钟的 CPU 计算时间（在现在的多核性能碾压下跑得飞快），反而贴进去了 3 分钟的网络下载和硬盘解压缩空转时间。
3. **触手 GitHub 免费天罚底线 (Eviction Death)**
   加上强大的 `Ccache` 之后（它可是个 5GB 级别的重量级），如果你再搞个 5GB 的巨无霸放在这里。分分钟撑破 GitHub 那可怜的 **10GB** 的天花板，到时候 GitHub 每隔几分钟就把你的缓存当作垃圾强制清空，你会遭遇 100% 的终极翻车。

## 👨‍💻 第三章：最佳实践与妥协的艺术 (Conclusion)

编程界有句名言：“过早的优化是一切罪恶的根源 (Premature optimization is the root of all evil)”。

**保持现在的姿态，是最优雅的平衡！**
在 `$(nproc) + 1` 及 `eatmydata` 无盘满载并发的算力加持下，`tools` 也就是你喝口水、点个外卖的几分钟功夫。
让系统在这几分钟里老老实实地只生产必要的工具，把省下的极其珍贵的几 GB 空间额度，让给我们在后面实打实提拔几十倍速度的 `Ccache` 和 `Toolchain_dir`。这就像是在一台跑车上，虽然有一个小刮痕看着不顺眼，但绝不能为此去装上几吨重的无用装甲，导致跑车被彻底压垮。
